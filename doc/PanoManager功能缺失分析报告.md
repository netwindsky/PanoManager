# PanoManager 后台管理系统 - 功能缺失分析与改进建议

**生成日期**: 2026-07-05  
**分析范围**: PanoManager 前端项目 + 后端 Spring Boot API  
**项目版本**: 当前开发分支

---

## 一、项目现状概览

### 1.1 已有功能模块

| 模块 | 路由 | 完成度 | 说明 |
|------|------|--------|------|
| 登录认证 | `/login` | ✅ 90% | 登录流程完整，缺少刷新token机制 |
| 数据概览 | `/dashboard` | ⚠️ 40% | 统计卡片有基础数据展示，缺少图表和趋势分析 |
| 项目管理 | `/projects` | ⚠️ 35% | 项目列表/新建/删除可用，编辑、发布、场景管理缺失 |
| 用户管理 | `/users` | ⚠️ 40% | 用户列表/启用禁用可用，新增用户、角色编辑、详情缺失 |
| 媒体资源 | `/media` | ⚠️ 25% | 资源列表/删除可用，上传、文件夹管理、预览缺失 |
| LUT管理 | `/luts` | ⚠️ 20% | LUT列表/删除可用，上传LUT、预览、分配缺失 |
| 后期预设 | `/presets` | ❌ 15% | 预设列表只读展示，创建、编辑、删除全部缺失 |
| 系统设置 | `/settings` | ❌ 10% | UI框架已搭建，保存/读取逻辑未完成 |
| 操作日志 | `/logs` | ⚠️ 50% | 日志列表可用，搜索过滤弱，详情查看缺失 |

**整体完成度评估**: 约 **28%** — 大部分页面只有基础UI框架和列表读取功能，CRUD操作不完整。

### 1.2 后端已支持但前端完全缺失的管理功能

| 功能模块 | 后端API支持情况 | 前端现状 |
|----------|----------------|----------|
| 场景管理 | 完整CRUD + 重新切片 | 完全没有管理页面 |
| 热点管理 | 完整CRUD | 完全没有管理页面 |

| LUT上传 | 文件上传接口 | UI按钮缺失 |
| 趋势统计 | 按天趋势数据API | Dashboard未接入 |

| 项目编辑 | 更新项目API | 前端无编辑入口 |

| Token刷新 | refresh接口 | 未实现token自动刷新 |

## 二、分模块详细问题分析

### 2.1 数据概览 (Dashboard)

**当前实现问题**:
- 只有4个统计卡片，项目总数/总浏览量/用户总数/已发布项目
- 统计数据来自 `/api/admin/stats`，但后端还返回 `storageUsed`（存储使用量）、`draftCount`（草稿数）未被使用
- 已定义 `ChartBar.vue` 和 `ChartPie.vue` 组件，但**未被使用**
- 缺少关键图表：
  - 近7/30天浏览量趋势图
  - 项目状态分布饼图
  - 最近活跃用户列表
  - 存储使用情况进度条
  - 最新项目列表
- 刷新按钮只刷新卡片数据，无其他联动

**后端可用但未使用的API**:
- `GET /api/stats/trend?days=N` — 获取趋势数据

**建议改进**:
1. 接入趋势API，添加近7天浏览量折线/柱状图
2. 添加项目状态分布饼图（草稿/已发布/已下架）
3. 添加存储使用量统计卡片
4. 添加"最近活动"或"待审核项目"区域
5. 考虑添加快捷操作入口

### 2.2 项目管理 (Projects)

**当前实现问题**:
- ✅ 项目列表分页加载正常
- ✅ 新建项目弹窗（名称+描述）可用
- ✅ 删除项目（有二次确认）可用
- ✅ 打开查看器/编辑器跳转链接可用
- ❌ "批量导出"按钮无点击事件 — 空壳
- ❌ 缺少项目搜索/关键词过滤功能（后端支持keyword参数）
- ❌ 无法编辑项目基本信息（名称、描述、封面）— 后端 `PUT /api/projects/{id}` 未调用
- ❌ 无法切换项目发布状态（发布/下架/草稿）
- ❌ 点击项目卡片只能查看/编辑/删除，**无法进入项目详情管理场景**
- ❌ 项目封面显示emoji，不显示真实封面图
- ❌ 缺少批量操作（批量删除、批量发布）
- ❌ 缺少按状态筛选（全部/草稿/已发布/已下架）
- ❌ 项目详情页不存在 — 无法管理项目下的场景

**核心缺失: 项目详情 → 场景管理**
这是最大的功能缺口。后端支持：
```
GET    /api/projects/{id}/scenes           # 列出项目下所有场景
POST   /api/projects/{id}/scenes           # 创建新场景
GET    /api/scenes/{id}                    # 获取场景详情
PUT    /api/scenes/{id}                    # 更新场景
DELETE /api/scenes/{id}                    # 删除场景
POST   /api/scenes/{id}/retiling           # 重新切片
GET    /api/scenes/{id}/tiling-progress    # 查询切片进度
```
前端完全没有场景管理相关的页面和API封装。管理员无法在后台查看/管理每个项目的场景。

**建议改进**:
1. 添加搜索框，支持按项目名称搜索（对接已有的keyword参数）
2. 添加项目状态筛选tabs
3. 实现项目编辑功能（名称/描述/封面上传）
4. 实现发布状态切换按钮
5. 点击项目卡片进入项目详情页，包含：
   - 项目基本信息编辑
   - 场景列表管理（添加/删除/编辑场景）
   - 场景切片进度显示
   - 场景缩略图预览
   - 跳转热点管理入口
6. 显示真实项目封面而不是emoji
7. 实现批量操作

// api/log.ts 中:
export function getLogs(params: PageParams) {

## 四、代码质量问题

### 4.1 已发现的Bug

| # | 文件 | 问题描述 | 严重程度 |
|---|------|----------|----------|
| 1 | `src/api/log.ts` | API路径错误：调用 `/admin/logs`，后端实际是 `/logs` | 🔴 高 |
| 2 | `src/views/LutView.vue` | 导入了`DataTable`和`TableColumn`，定义了`columns`但模板中完全没使用DataTable，用的是自定义列表 | 🟡 中 |
| 3 | `src/views/SettingsView.vue` | `getSettings()` 返回类型是 `List<SystemConfig>` 不是 `Record<string, string>`，类型不匹配，解析会失败 | 🟡 中 |
| 4 | `src/api/lut.ts` | 使用 `/public/luts` 公开接口，管理后台应该有管理接口 | 🟡 中 |
| 5 | 各页面 | 大量按钮无点击事件处理（空壳按钮），用户点击无反应 | 🟡 中 |
| 6 | 各页面catch块 | 大量 `catch {}` 空catch块，错误被静默吞噬，出问题无法排查 | 🟡 中 |
| 7 | `src/views/LogsView.vue` | 传keyword参数给后端，但后端日志API不支持keyword，搜索无效 | 🟡 中 |

### 4.2 架构/代码风格问题

1. **UI库不统一**:
   - PanoEditor使用Element Plus
   - PanoManager大部分组件是自己手写CSS，但LoginView和一些地方又有混入
   - DataTable、Pagination等组件是自定义的，质量不高，维护成本大
   - **建议**: 统一使用Element Plus组件库，减少自定义组件维护成本

2. **类型定义问题**:
   - `UserRole` 类型只定义了 `'ADMIN' | 'VIEWER'`，但代码中出现了 `'editor'` 角色（UserMenu中roleMap有editor）
   - 多处使用 `ref<Record<string, unknown>[]>` 代替正确的类型，丢失了类型安全
   - ProjectStatus类型定义了但没有被使用，状态字符串硬编码

3. **API层问题**:
   - 所有catch块都是空的 `catch {}` 或 `catch { // 忽略 }`，错误无法被追踪
   - 没有统一的错误提示，API失败时用户没有感知（除了LoginView）
   - 没有请求loading状态管理
   - 分页参数类型重复定义，没有统一复用

4. **组件复用问题**:
   - ChartBar、ChartPie组件写好了但没被任何页面使用
   - DataTable组件接收columns和data，但很多页面没有使用它，自己写列表
   - StatCard组件使用正常，但缺少图表组件配套

5. **状态管理问题**:
   - stores只有 `app.ts` 和 `auth.ts` 两个
   - app.ts 几乎没被使用（只有Sidebar折叠状态？）
   - 数据获取逻辑都在组件内，没有封装到store或composables
   - 没有缓存机制，切换页面重新加载所有数据

6. **样式问题**:
   - 全局CSS变量定义在 `styles/variables.css`，需要确认是否在main.ts中引入
   - 大量重复的样式代码（每个页面都重复写 `.page-header`, `.btn`, `.loading-state` 等）
   - 没有提取公共样式class

7. **缺少基础设施**:
   - 没有消息提示/Toast组件（删除/创建成功/失败用户无感知）
   - 没有确认对话框组件（删除确认用的是原生 `confirm()`）
   - 没有表单验证
   - 没有文件上传组件
   - 没有加载进度条组件

8. **安全问题**:
   - 401直接跳转没有token刷新，用户在填写长表单时token过期会丢失所有输入
   - 没有操作权限控制（前端路由只检查是否登录，不检查角色，所有登录用户都能看到所有管理页面）
   - 删除操作需要更安全的确认机制

## 八、总结

PanoManager 作为全景平台的**管理后台**，目前完成度约**28%**。基础框架搭起来了，9个页面的UI骨架都有，列表读取功能基本可用，但**所有写操作（增删改）除了删除项目/用户/资源/LUT/日志查看，其他大量功能都是空壳按钮**。

最大的功能缺口是：**管理员无法在后台管理场景和热点**。场景和热点是全景平台的核心业务实体，后端API已经完整支持，但前端完全没有对应管理页面，只能通过PanoEditor编辑器操作。这意味着当前后台无法审核/修改/删除具体场景和热点内容，管理能力严重不足。

其次是所有页面的"新增"、"编辑"、"导出"按钮普遍是空壳，点击无反应，用户体验断层。

建议按优先级分阶段实施，先修复bug和补全现有页面的基础CRUD，再推进场景/热点核心管理页面的开发。
